Quickstart (TensorFlow 2)¶

Before You Start¶

We recommend you first finish either the Quickstart (PyTorch) or the Quickstart (Numpy) exercise. Those guides go more in depth in explaining the federated learning aspect of NVFlare.

Here we assume you have already installed NVFlare inside a python virtual environment and have already cloned the repo.

Introduction¶

Through this exercise, you will integrate NVFlare with the popular deep learning framework TensorFlow 2 and learn how to use NVFlare to train a convolutional network with the MNIST dataset. You will also be introduced to some new components and concepts, including the filter, aggregrator, and event handler.

The design of this exercise consists of one server and two clients all having the same TensorFlow 2 model. The following steps compose one cycle of weight updates, called a round:

  1. Clients are responsible for generating individual weight-updates for the model using their own MNIST dataset.

  2. These updates are then sent to the server which will aggregate them to produce a model with new weights.

  3. Finally, the server sends this updated version of the model back to each client.

For this exercise, we will be working with the hello-tf2 application in the examples folder. Every custom FL application must contain three folders:

  1. custom: contains the custom components (net.py, trainer.py, filter.py, model_aggregator.py, model_persistor.py)

  2. config: contains client and server configurations (config_fed_client.json, config_fed_server.json)

  3. resources: contains the logger config (log.config)

Let’s get started. Since this task is using TensorFlow, let’s go ahead and install the library inside our virtual environment:

(nvflare-env) $ python3 -m pip install tensorflow

NVFlare Client¶

Neural Network¶

With all the required dependencies installed, you are ready to run a Federated Learning with two clients and one server. Before you start, let’s see what a simplified MNIST network looks like.

net.py¶
 1import tensorflow as tf
 2
 3
 4class Net(tf.keras.Model):
 5    def __init__(self):
 6        super().__init__()
 7        self.flatten = tf.keras.layers.Flatten(input_shape=(28, 28))
 8        self.dense1 = tf.keras.layers.Dense(128, activation="relu")
 9        self.dropout = tf.keras.layers.Dropout(0.2)
10        self.dense2 = tf.keras.layers.Dense(10)
11
12    def call(self, x):
13        x = self.flatten(x)
14        x = self.dense1(x)
15        x = self.dropout(x)
16        x = self.dense2(x)
17        return x

This Net class is the convolutional neural network to train with MNIST dataset. This is not related to NVFlare, so implement it in a file called net.py.

Dataset & Setup¶

Now you have to implement the only class related to NVFlare, Trainer, in a file called trainer.py.

Before you can really start a training, you need to setup your dataset. In this exercise, you can download it from the Internet via tf.keras’s datasets module, and split it in half to create a separate dataset for each client. Additionally, you must setup the optimizer, loss function and transform to process the data.

Since every step will be encapsulated in the Trainer class, let’s put this preparation stage into one method setup:

24    def setup(self, fl_ctx):
25        (self.train_images, self.train_labels), (
26            self.test_images,
27            self.test_labels,
28        ) = tf.keras.datasets.mnist.load_data()
29        self.train_images, self.test_images = (
30            self.train_images / 255.0,
31            self.test_images / 255.0,
32        )
33
34        # simulate separate datasets for each client by dividing MNIST dataset in half
35        client_name = fl_ctx.get_prop(FLConstants.CLIENT_NAME)
36        if client_name == "site-1":
37            self.train_images = self.train_images[:len(self.train_images)//2]
38            self.train_labels = self.train_labels[:len(self.train_labels)//2]
39            self.test_images = self.test_images[:len(self.test_images)//2]
40            self.test_labels = self.test_labels[:len(self.test_labels)//2] 
41        elif client_name == "site-2":
42            self.train_images = self.train_images[len(self.train_images)//2:]
43            self.train_labels = self.train_labels[len(self.train_labels)//2:]
44            self.test_images = self.test_images[len(self.test_images)//2:]
45            self.test_labels = self.test_labels[len(self.test_labels)//2:] 
46        
47        model = Net()
48
49        loss_fn = tf.keras.losses.SparseCategoricalCrossentropy(from_logits=True)
50        model.compile(optimizer="adam", loss=loss_fn, metrics=["accuracy"])
51        _ = model(tf.keras.Input(shape=(28, 28)))
52
53        self.model = model

How can you ensure this setup method is called before the client receives the model from the server? The Trainer class is also a FLComponent, which always receives Event whenever NVFlare enters or leaves a certain stage. In this case, there is an Event called EventType.START_RUN which perfectly matches these requirements. Because our trainer is a subclass of FLComponent, you can implement the handler to handle the event and call the setup method:

20    def handle_event(self, event_type: str, fl_ctx: FLContext):
21        if event_type == EventType.START_RUN:
22            self.setup(fl_ctx)

Note

This is a new concept you haven’t learned in previous two exercises. The concepts about event and handler are very powerful because you are free to add your logic so it can run at different time and process various events. The entire list of events fired by NVFlare is shown at Event types.

You have everything you need, now let’s implement the last method called train, which is called every time the client receives an updated model from the server.

Link NVFlare with Local Train¶

Take a look at the following code:

 55    def train(self, shareable: Shareable, fl_ctx: FLContext, abort_signal) -> Shareable:
 56        """
 57        This function is an extended function from the super class.
 58        As a supervised learning based trainer, the train function will run
 59        evaluate and train engines based on model weights from `shareable`.
 60        After finishing training, a new `Shareable` object will be submitted
 61        to server for aggregation.
 62
 63        Args:
 64            shareable: the `Shareable` object acheived from server.
 65            fl_ctx: the `FLContext` object achieved from server.
 66            abort_signal: if triggered, the training will be aborted.
 67
 68        Returns:
 69            a new `Shareable` object to be submitted to server for aggregation.
 70        """
 71
 72        # retrieve model weights download from server's shareable
 73        model_weights = shareable[ShareableKey.MODEL_WEIGHTS]
 74
 75        # use previous round's client weights to replace excluded layers from server
 76        prev_weights = {
 77            self.model.get_layer(index=key).name: value for key, value in enumerate(self.model.get_weights())
 78        }
 79        for key, value in model_weights.items():
 80            if np.all(value == 0):
 81                model_weights[key] = prev_weights[key]
 82
 83        # update local model weights with received weights
 84        self.model.set_weights(list(model_weights.values()))
 85
 86        # adjust LR or other training time info as needed
 87        # such as callback in the fit function
 88        self.model.fit(
 89            self.train_images,
 90            self.train_labels,
 91            epochs=self.epochs_per_round,
 92            validation_data=(self.test_images, self.test_labels),
 93        )
 94
 95        # report updated weights in shareable
 96        shareable = Shareable()
 97        shareable[ShareableKey.TYPE] = ShareableValue.TYPE_WEIGHTS
 98        shareable[ShareableKey.DATA_TYPE] = ShareableValue.DATA_TYPE_UNENCRYPTED
 99        shareable[ShareableKey.MODEL_WEIGHTS] = {
100            self.model.get_layer(index=key).name: value for key, value in enumerate(self.model.get_weights())
101        }
102        self.logger.info(f"Sending shareable to server: \n {shareable[ShareableKey.MODEL_WEIGHTS]}")
103        return shareable

Every NVFlare client receives the model weights from the server in the shareable. This exercise uses a simple exclude_var filter, so make sure to replace the missing layer with weights from the clients’ previous training round:

for key, value in model_weights.items():
      if np.all(value == 0):
          model_weights[key] = prev_weights[key]

Now update the local model with those received weights:

self.model.set_weights(list(model_weights.values()))

Then peform a simple self.model.fit so the client’s model is trained with its own dataset:

self.model.fit(
    self.train_images,
    self.train_labels,
    epochs=self.epochs_per_round,
    validation_data=(self.test_images, self.test_labels),
)

After finishing the local train, the train method builds a new shareable with newly-trained weights and returns it back to the NVFlare server.

NVFlare Server & Application¶

Filter¶

The filter is used for additional data processing in the shareable, either inbound or outbound from the client and/or server.

For this exercise, we use a basic exclude_var filter to exclude the variable/layer flatten outbound from the client to the server. The excluded layer is replaced with all zeros of the same shape, which reduces compression size and ensures that the clients’ weights for this variable are not shared with the server.

filter.py¶
 1import numpy as np
 2
 3from nvflare.apis.filter import Filter
 4from nvflare.apis.fl_constant import FLConstants, ShareableKey, ShareableValue
 5from nvflare.apis.fl_context import FLContext
 6from nvflare.apis.shareable import Shareable
 7
 8
 9class ExcludeVars(Filter):
10    """
11        Exclude/Remove variables from Sharable
12
13    Args:
14        exclude_vars: if not specified (None), all layers are being encrypted;
15                      if list of variable/layer names, only specified variables are excluded;
16    """
17
18    def __init__(self, exclude_vars=None):
19        super().__init__()
20        self.exclude_vars = exclude_vars
21
22        if self.exclude_vars is not None:
23            assert isinstance(self.exclude_vars, list), (
24                "Provide a list of layer names "
25            )
26            for var in self.exclude_vars:
27                assert isinstance(var, str), (
28                    "encrypt_layers needs to be a list of layer names to encrypt."
29                )
30            self.logger.info(f"Excluding {self.exclude_vars} from shareable")
31        else:
32            self.logger.info(f"Not excluding anything")
33
34    def process(self, shareable: Shareable, fl_ctx: FLContext) -> Shareable:
35        weights = shareable[ShareableKey.MODEL_WEIGHTS]
36
37        # set excluded variables to all zeros
38        n_excluded = 0
39        var_names = list(weights.keys()) 
40        n_vars = len(var_names)
41        for var_name in var_names:
42            if self.exclude_vars:
43                if var_name in self.exclude_vars:
44                    self.logger.info(f"Excluding {var_name}")
45                    weights[var_name] = np.zeros(weights[var_name].shape)        
46                    n_excluded += 1
47        self.logger.info(f"Excluded {n_excluded} of {n_vars} variables")
48
49        shareable[ShareableKey.MODEL_WEIGHTS] = weights
50        return shareable

The filtering procedure occurs in the one required method, process, which receives and returns a shareable. The parameters for what is excluded and the inbound/outbound option are all set in config_fed_client.json (shown later below) and passed in through the constructor.

Model Aggregator¶

The model aggregator is used by the server to aggregrate the clients’ models into one model.

model_aggregator.py¶
 1from typing import Tuple
 2import numpy as np
 3
 4from nvflare.apis.aggregator import Aggregator
 5from nvflare.apis.fl_constant import FLConstants, ShareableKey, ShareableValue
 6from nvflare.apis.fl_context import FLContext
 7from nvflare.apis.shareable import Shareable
 8from nvflare.utils.fl_ctx_sanity_check import server_fl_ctx_sanity_check
 9
10
11class AccumulateAggregator(Aggregator):
12    def __init__(self, exclude_vars=None, aggregation_weights=None):
13        super().__init__()
14        self.accumulator = []
15
16    def accept(self, shareable: Shareable, fl_ctx: FLContext) -> Tuple[bool, bool]:
17        """Store shareable and update aggregator's internal state
18        Args:
19            shareable: information from client
20            fl_ctx: context provided by workflow
21
22        Returns:
23            The first boolean indicates if this shareable is accepted.
24            The second boolean indicates if aggregate can be called.
25        """
26
27        server_fl_ctx_sanity_check(fl_ctx)
28        current_round = fl_ctx.get_prop(FLConstants.CURRENT_ROUND)
29        shared_fl_context = fl_ctx.get_prop(FLConstants.PEER_CONTEXT)
30        client_name = shared_fl_context.get_prop(FLConstants.CLIENT_NAME)
31        contribution_round = shared_fl_context.get_prop(FLConstants.CURRENT_ROUND)
32
33        accepted = False
34        if contribution_round == current_round and not self._client_in_accumulator(client_name):
35            self.accumulator.append(shared_fl_context)
36            accepted = True
37
38        return accepted, False
39
40    def _client_in_accumulator(self, client_name):
41        for item in self.accumulator:
42            if client_name == item.get_prop(FLConstants.CLIENT_NAME):
43                return True
44        return False
45
46    def aggregate(self, fl_ctx: FLContext) -> Shareable:
47        """Called when workflow determines to generate shareable to send back to clients
48
49        Args:
50            fl_ctx (FLContext): context provided by workflow
51
52        Returns:
53            Shareable: the average of accepted shareables from clients
54        """
55        
56        server_fl_ctx_sanity_check(fl_ctx)
57        current_round = fl_ctx.get_prop(FLConstants.CURRENT_ROUND)
58        self.logger.info("aggregating %s updates at round %s", len(self.accumulator), current_round)
59
60        vars_to_aggregate = self.accumulator[0].get_prop(FLConstants.SHAREABLE)[ShareableKey.MODEL_WEIGHTS].keys()
61
62        # perform average of clients' weights
63        aggregated_model = {}
64        for v_name in vars_to_aggregate:
65            np_vars = []
66            for client_ctx in self.accumulator:
67                data = client_ctx.get_prop(FLConstants.SHAREABLE)[ShareableKey.MODEL_WEIGHTS]
68                np_vars.append(data[v_name])
69
70            new_val = np.sum(np_vars, axis=0) / len(self.accumulator)
71            aggregated_model[v_name] = new_val
72
73        self.accumulator.clear()
74
75        shareable = Shareable()
76        shareable[ShareableKey.TYPE] = ShareableValue.TYPE_WEIGHTS
77        shareable[ShareableKey.DATA_TYPE] = ShareableValue.DATA_TYPE_UNENCRYPTED
78        shareable[ShareableKey.MODEL_WEIGHTS] = aggregated_model
79        return shareable
80

In this exercise, we perform a simple average over the two clients’ weights.

The accept() function is responsible for managing the shareables from the clients. Note that the second return boolean can be set to True based on some logic to call aggregate() early, otherwise by default aggregate() will always be called once the number of submissions is >= min_clients.

The aggregrate() function then performs the aggregration procedure and returns a new shareable.

FLContext is used throughout these functions to provide various useful FL-related information. You can find more details in the documentation.

Model Persistor¶

The model persistor is used to load and save models on the server.

model_persistor.py¶
 1import os
 2import pickle
 3import numpy as np
 4
 5import tensorflow as tf
 6from net import Net
 7from nvflare.apis.fl_context import FLContext
 8from nvflare.apis import Model, ModelPersistor
 9from nvflare.apis.fl_constant import FLConstants
10
11
12class TF2ModelPersistor(ModelPersistor):
13    def __init__(self, save_name=None):
14        super().__init__()
15        self.save_name = save_name
16
17    def _initialize(self, fl_ctx: FLContext):
18        # get save path from FLContext
19        train_root = fl_ctx.get_prop(FLConstants.TRAIN_ROOT)
20        log_dir = fl_ctx.get_prop(FLConstants.LOG_DIR)
21        if log_dir:
22            self.log_dir = os.path.join(train_root, log_dir)
23        else:
24            self.log_dir = train_root
25        self._pkl_save_path = os.path.join(self.log_dir, self.save_name)
26        if not os.path.exists(self.log_dir):
27            os.makedirs(self.log_dir)
28
29    def load_model(self, fl_ctx: FLContext) -> Model:
30        """
31            initialize and load the Model.
32
33        Args:
34            fl_ctx: FLContext
35
36        Returns:
37            Model object
38        """
39        self._initialize(fl_ctx)
40
41        model = Model()
42        if os.path.exists(self._pkl_save_path):
43            self.logger.info(f"Loading server weights")
44            with open(self._pkl_save_path, "rb") as f:
45                var_dict = pickle.load(f)
46        else:
47            self.logger.info(f"Initializing server model")
48            network = Net()
49            loss_fn = tf.keras.losses.SparseCategoricalCrossentropy(from_logits=True)
50            network.compile(optimizer="adam", loss=loss_fn, metrics=["accuracy"])
51            _ = network(tf.keras.Input(shape=(28, 28)))
52            var_dict = {
53                network.get_layer(index=key).name: value
54                for key, value in enumerate(network.get_weights())
55            }
56        model.update(var_dict)
57        return model
58
59    def save_model(self, model: Model, fl_ctx: FLContext):
60        """
61            persist the Model object
62
63        Args:
64            model: Model object
65            fl_ctx: FLContext
66        """
67        self.logger.info(f"Saving aggregated server weights: \n {model}")
68        with open(self._pkl_save_path, "wb") as f:
69            pickle.dump(model, f)

In this exercise, we simply serialize the model weights dictionary using pickle and save it to server/run_1/mmar_server/models/tf2weights.pickle inside the server directory (weights file name defined in config_fed_server.json). Depending on the frameworks and tools, the methods of saving the model may vary.

Train the Model, Federated!¶

Now you must set up a local environment and generate packages to simulate the server, clients, and admin.

Setting Up the Application Environment¶

This command generates a poc folder with server, one client and one admin:

$ poc

Here we duplicate the client folder into two folders to create two clients, site-1 and site-2:

$ cp -r poc/client poc/site-1
$ cp -r poc/client poc/site-2

Finally, we copy necessary files (the exercise codes) to a working folder:

$ mkdir -p poc/admin/transfer
$ cp -rf examples/* poc/admin/transfer

With both the client and server ready, you can now run everything and see federated learning in action. FL systems usually have a server and multiple clients. We therefore have to start the server first:

$ ./poc/server/startup/start.sh

Once the server is running you can start the clients in different terminals. Open a new terminal and start the first client:

$ ./poc/site-1/startup/start.sh site-1 localhost

Open another terminal and start the second client:

$ ./poc/site-2/startup/start.sh site-2 localhost

In one last terminal, start the admin:

$ ./poc/admin/startup/fl_admin.sh localhost

This will launch a command prompt, where you can input commands to control and monitor many aspects of the FL process. Log in by entering admin for both the username and password.

Running the FL¶

Enter the commands below in order. Pay close attention to what happens in each of four terminals. You can see the admin controls server and clients with each command.

> upload_app hello-tf2

Uploads an application in the server’s registry. This creates the application entry, populates the configuration and links the name hello-tf2 with such application configuration. Later, you can control this application via this name.

> set_run_number 1

Creates a workspace with the run_number on the server and all clients. The purpose of the workspace is to isolate different runs so the information in one particular run does not interfere with other runs.

> deploy_app hello-tf2 all

This will make the hello-tf2 application the active one in the run_number workspace. In this exercise, after the above two commands, the server and all the clients know the hello-tf2 application will reside in run_1 workspace.

> start_app all

This start_app command instructs the NVFlare server and clients to start training with the hello-tf2 application in that run_1 workspace.

From time to time, you can issue check_status server in the admin client to check the entire training progress.
